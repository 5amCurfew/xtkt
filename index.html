Overview

xtkt ("extract") is a generic but opinionated data extraction tool that adheres to the Singer.io specification. At its core, xtkt takes an opinionated approach to data extraction (ELT for OLAP) importing new and updated data as new rows when a bookmark is provided using either new-record detection or an appropriate bookmark for any RESTful API. Streams are always handled independently and deletion at source is not detected. xtkt can be pipe'd to any target that meets the Singer.io specification but has been designed and tested for databases such as Postgres and BigQuery.

💡 In the absence of a bookmark (referred to as FULL_TABLE replication), all records (new, updated and unchanged) will be inserted or updated per unique_key on each xtkt invocation. This may be appropriate when inferring deletion using the time_extracted field.

Metadata

xtkt adds the following metadata to records


  • surrogate_key: an identifier of a record (SHA256) generated using the unique_key and bookmark (if provided). If record detection is used, this is generated using the entire record object

  • time_extracted: a timestamp (R3339) at the time of the data extraction

    • Usage

    >> xtkt <RELATIVE_PATH_TO_CONFIG_JSON> | your-target

    💡 example: xtkt config.json | ./_targets/pipelinewise-target-postgres/bin/target-postgres -c postgres.json


    config.json


  • url: string the Restful API address
  • auth.required: boolean stating if an authentication method is used
  • response.pagination: boolean stating if response pagination is used
  • records.unique_key_path: array stating relative path within a response record for the unique identifier of a record
  • records.bookmark: boolean stating whether state should be tracked using a bookmark

    • Optional


    auth

  • auth.strategy: string stating authorisation strategy (accepted values are currently "basic" or "token")
  • auth.basic: {object} objecting containing credentials for basic authorisation {"username", "password"}
  • auth.token: {object} objecting containing request Header information for token authorisation {"header", "header_value"}

    • response

  • response.records_path: array stating relative path for records in response
  • response.pagination_strategy: string stating pagination strategy (accepted values are current "next")
  • response.pagination_next_path: array stating relative path within response for the next page address

    • records

  • records.primary_bookmark_path: array stating relative path within record for bookmark value (to use new/updated record detection use ["*"]

    • 💡 record detection is not advised for large data sources

    example

              
{
  "url": "https://www.helloworld.com/route",
  "auth": {
      "required": true,
      "strategy": "token",
      "basic": {
          "username": "u",
          "password": "p"
      },
      "token": {
          "header": "Authorization",
          "header_value": "Bearer YOUR_API_TOKEN"
      }
  },
  "response": {
      "records_path": [
          "results"
      ],
      "pagination": true,
      "pagination_strategy": "next",
      "pagination_next_path": [
          "info",
          "next"
      ],
      "pagination_query": {
          "query_parameter": "page",
          "query_value": 1,
          "query_increment": 1
      }
  },
  "records": {
      "unique_key_path": [
          "id"
      ],
      "bookmark": true,
      "primary_bookmark_path": [
          "updated_at"
      ]
  }
}

    example config.json

                        
{
  "url": "https://rickandmortyapi.com/api/character",
  "auth": {
      "required": false
  },
  "response": {
      "records_path": [
          "results"
      ],
      "pagination": true,
      "pagination_strategy": "next",
      "pagination_next_path": [
          "info",
          "next"
      ]
  },
  "records": {
      "unique_key_path": ["id"],
      "bookmark": true,
      "primary_bookmark_path": ["*"]
  }
}
